# SPEC Subagent Review — v2 (brand-assets-optimization)
Reviewer: read-only subagent · Date: 2026-06-14

## Verdict
APPROVE WITH CHANGES — the contracts are internally consistent, decision-aligned (posters all-true, Gemini gated), and the discriminated-union / plan / icon-derive signatures are implementable against current code; two real gaps must be fixed before PLAN: (1) the strict+superRefine product schema cannot take `.optional()` the way SPEC-DATA-002's comment implies — the field must go inside `.extend({...})` before `.strict()`; (2) SPEC-DATA-008's test-deletion list omits `brand-assets.test.ts:31,630`, a dangling `STORE_SHOT_PRESETS` import that will break typecheck.

## Blocking Findings

1. **SPEC-DATA-002 product-schema extension is mis-described — `.optional()` cannot be tacked onto the strict+superRefine schema.** `productSchema` (product.ts:53-112) is `productIndexEntrySchema.extend({...}).strict().superRefine(...)`. After `.strict().superRefine(...)` it is a `ZodEffects`, which has **no `.extend()` / object-merge surface**. The SPEC comment `// productSchema (.strict() chain): + brand_assets: brandAssetsSettingsSchema.optional()` reads as if `brand_assets` is appended to the chain; that won't typecheck. Fix: PLAN must add `brand_assets: brandAssetsSettingsSchema.optional()` **inside the existing `.extend({...})` object (product.ts:54-63), before `.strict()`**. The new optional block does not interact with the `superRefine` (lines 65-112), so no superRefine change is needed — but the SPEC should state this explicitly so the impl does not append after the chain. (This is exactly Advisory-1 of design-review-v3, here promoted to BLOCKING because SPEC-DATA-002's own comment encodes the wrong placement.)

2. **SPEC-DATA-008 test-deletion list omits `brand-assets.test.ts:31,630` (`STORE_SHOT_PRESETS` import + usage).** SPEC-DATA-008's "测试" bullet lists only `store-shot-presets.test.ts` (whole file) and `tools.test.ts:6668-6790 + :277`. But `packages/core/tests/brand-assets.test.ts:31` imports `STORE_SHOT_PRESETS` and uses it at `:630` (`const preset = STORE_SHOT_PRESETS["web-og"]`). When SPEC-BEHAVIOR-007 removes `STORE_SHOT_PRESETS`, this file fails `pnpm typecheck` — TM-09 would catch it at runtime, but SPEC-DATA-008 is declared "PLAN 删除清单的权威来源" and is incomplete. Fix: add `packages/core/tests/brand-assets.test.ts:31` (import) + `:630` (usage) to SPEC-DATA-008's test bullet. (design-review-v3 already enumerated this in its sweep; the SPEC dropped it.)

## Advisory Findings

1. **SPEC-DATA-006 changes `saveBrandAsset`'s return type for the app-icon branch — call out the union return explicitly.** Current `saveBrandAsset(deps, input): Promise<SavedBrandAsset>` (brand-assets.ts:585) returns a single record's `SavedBrandAsset`. SPEC-DATA-006 introduces `SavedAppIconResult = { assets: BrandAssetRecord[] }` for `kind:"app-icon"`. The SPEC types the inputs as a discriminated union but does not pin the **return** union (`SavedBrandAsset` for html kinds vs `SavedAppIconResult` for app-icon). PLAN should specify the return contract (e.g. `Promise<SavedBrandAsset | SavedAppIconResult>` or split functions) so the MCP `save_brand_asset` handler and `{assets:[...]}` shape (AC-004) are unambiguous.

2. **`resolveBrandImageRef` bare-ref `MASTER_SIZE=2048` selection breaks once the 2048 master is gone.** brand-assets.ts:747 keys the bare `forma-image://brand/app-icon` ref to `width === MASTER_SIZE (2048)`. The new pipeline normalizes to a 1080 canvas with largest standard = 1024/512; no 2048 file will exist, so the bare ref throws `size_not_in_set`. SPEC-BEHAVIOR-005/006 remove the `name==="primary"` fallback but do not specify the **new bare-ref pixel-selection rule** (e.g. "largest `variant==="standard"` file"). PLAN must define it, and TM should add a bare-ref resolution assertion. (Matches design-review-v3 Advisory-4 / Coverage gap; SPEC did not absorb it.)

3. **`ProductConfigField` should be resolved to "not extended".** SPEC-DATA-002 does not mention `ProductConfigField` (product.ts:136 = `"platform"|"brand_style"|"languages"`), which is correct under D9 (brand-asset settings live on a separate always-editable panel with `updateBrandAssetSettings`, not the one-time config form). No change needed — flagging only so PLAN does not "helpfully" extend it.

4. **SPEC-DATA-002 / SPEC-DATA-007 settings-write path: no MCP write surface.** SPEC-DATA-007 correctly adds only read (`get_brand_asset_plan`) + delete; `updateBrandAssetSettings` is web/server-only per D9 ("不开放 MCP 写"). Consistent — no finding, confirming for closure.
