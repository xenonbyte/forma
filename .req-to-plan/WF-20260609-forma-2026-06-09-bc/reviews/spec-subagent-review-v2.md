## Findings
1. **[minor] DESIGN↔SPEC `shape` field-name divergence is silent.** DES-ARCH-004 names the sub-object `{ geometryId/shapeId/sourceVersion }` (05-design.md:67); SPEC-DATA-001 renames `geometryId`→`geometry` (now markup body) and keeps `shapeId/sourceVersion` (SPEC:71). This is the intended v4-#4 resolution and is *better*, but the spec never states it is renaming a DESIGN field, so a reader diffing the two docs sees an unexplained mismatch. *Fix:* add one clause to SPEC-DATA-001 noting it supersedes the DESIGN's `geometryId` identifier with a `geometry` body field per the delegated decision. Non-blocking.

2. **[minor] "约28件" is intentionally fuzzy but the spec also says "逐字镜像 … 不得改写".** SPEC-DATA-003 (SPEC:91) requires components to "逐字对应 raw-requirement B2 清单 (web 6 组约28件)" with "单测逐项断言齐全", while the source list (05-design.md:721-732) is itself approximate ("约 28 件", several rows bundle variants e.g. Button "× 尺寸 × 带图标"). The "verbatim mirror + per-item assertion" instruction is slightly in tension with an approximate, variant-bundling source. *Fix for PLAN:* PLAN must fix the exact enumerated set (e.g. count-locked group→component list) so the unit test has a deterministic expected array; treat the 6 groups + the explicit mobile-variant swaps as the authoritative shape, and pin a concrete count rather than "≈28". Non-blocking but decidability-relevant.

3. **[nit] `mapArtifactsToViewerInputs` location.** SPEC-BEHAVIOR-015 / SPEC-DATA-006 speak of a "product-level mapper" not reusing the design-page-premised `mapArtifactsToViewerInputs`. That function lives at `packages/web/src/viewer/mapArtifacts.ts:27` (web package), not in `packages/viewer`. The spec doesn't misstate this, but PLAN should place the new `brand-resources` mapper alongside it in `packages/web/src/viewer/` (with its own `mapArtifacts.test.ts` sibling) rather than in the `viewer` package. Nit.

4. **[nit] External-Documentation row is honest.** The @xyflow/react row (SPEC:107-112) makes no version-specific API claim, defers custom-node/selection/viewport API verification to Context7 at implementation time, and asserts only engine-agnostic behavioral contracts for BC2/BC3. This is appropriate — the spec correctly does not fabricate version-pinned API facts (`无 UNCONFIRMED 断言`). No action.
