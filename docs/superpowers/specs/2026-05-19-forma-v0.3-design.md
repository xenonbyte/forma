# Forma v0.3 Design

Date: 2026-05-19
Source: `design-version/DESIGN-v3.md`
Status: Approved design for implementation planning

## Goal

Implement Forma v0.3 as one complete release batch. The release adds product language configuration, unified requirement saving, structured multilingual UI copy, BDD logic rules, change-type aware design regeneration, Chinese/English Web admin localization, richer MCP data access, updated Agent commands, and refreshed project documentation.

The implementation follows the full scope in `design-version/DESIGN-v3.md`; no feature area from that source design is intentionally deferred.

## Scope

v0.3 includes:

- Product-level `languages` and `default_language` fields.
- Required platform, style, languages, and default language configuration during Web product creation.
- Language completeness checks in product session and design-generation gates.
- A unified `save_requirement` MCP and HTTP contract that replaces `submit_requirement` and `update_requirement`.
- A unified `fm-requirement` Agent command that accepts full PRDs, partial changes, feature bullets, or one-line changes.
- Requirement document formatting, page extraction, structured copy extraction, translations, BDD rule generation, change-type detection, and conflict review handled by Agent-side AI.
- Requirement-level `ui_affected` for logic-only changes that do not enter the design flow.
- Requirement page `change_type` and `change_summary` fields for `new`, `patch`, and `rebuild` page changes.
- Structured `copy: CopyItem[]` on requirement pages and baseline pages.
- Requirement-level `copy-translations.yaml` for non-default-language translations.
- Product baseline `rules.yaml` for accumulated BDD rules.
- New MCP tools for product rules and multilingual copy.
- Extended `get_requirement` and `help` MCP responses for AI developer workflows.
- Patch/rebuild support in `DesignService` using existing design history and revised `save_designs` mode gates.
- Web admin `en` / `zh` localization with local persistence.
- Web product config completion, empty requirement creation, and multilingual copy display.
- Agent command template updates for Claude, Codex, and Gemini.
- README, MCP, and Agent command documentation updates.

v0.3 does not include:

- Runtime design-language switching; generated designs use `default_language`.
- Web admin locales beyond English and Simplified Chinese.
- Automatic backfill for existing product language config.
- Version history for copy translations beyond requirement history.
- A data migration tool. Current Forma home data is assumed non-production, and schema-breaking changes are allowed.
- Backend semantic validation of requirement text. AI validation remains in Agent commands.
- A formal logic engine for BDD conflicts. Conflict detection is AI-assisted and user-confirmed.

## Architecture

The implementation is contract-first and keeps the existing monorepo boundaries.

| Package | v0.3 Responsibility |
| --- | --- |
| `packages/core` | Own schemas, persistence, service state machines, rules/copy storage, transaction rollback, product config checks, baseline updates, and design save gates. |
| `packages/mcp` | Expose the new stable MCP contract, aggregate cross-service developer-facing data, remove old requirement tools, and return structured errors. |
| `packages/server` | Expose thin Fastify routes for product config, empty requirement creation, requirement saving, and updated baseline/design metadata access. |
| `packages/web` | Consume HTTP APIs, implement admin localization, product configuration forms, empty requirement creation, and structured copy display. |
| `packages/agent` | Install the updated command set and platform-specific templates that drive AI-side requirement processing. |
| `docs` / `README.md` | Document the new MCP tools, Agent commands, architecture, and developer data retrieval workflow. |

`packages/core` is the source of truth for data contracts and state transitions. MCP and HTTP adapters do not duplicate business rules. Web and Agent templates consume those contracts and avoid direct filesystem assumptions.

## Core Data Contracts

`packages/core/src/schemas.ts` adds:

- `languages = ["zh-CN", "zh-TW", "en", "ja", "ko", "pt", "fr", "de", "ru"]`.
- `Language`.
- `CopyItem` shape through a shared schema used by requirements and baselines.

`Product` gains:

- `languages?: Language[]`.
- `default_language?: Language`.

The product schema enforces that both fields are present together, that `languages` has at least one value when present, and that `default_language` is included in `languages`.

`ProductConfig` becomes:

- `platform`.
- `style`.
- `languages`.
- `default_language`.

`assertProductConfig` moves from the MCP tools module into `packages/core/src/product.ts` and becomes a shared exported helper. `packages/core/src/session.ts` and `packages/mcp/src/tools.ts` both use this helper so product session checks and design-generation gates cannot drift.

`Requirement` gains:

- `ui_affected: boolean`, defaulting to `true` for empty placeholders.

`RequirementPage` gains:

- `copy?: CopyItem[]`, replacing the old string field.
- `change_type?: "new" | "patch" | "rebuild"`.
- `change_summary?: string`.

`BaselinePage.copy` also becomes `CopyItem[]`.

## Requirement Save State Machine

`RequirementService.saveRequirement(input)` becomes the only requirement content write path. Existing `submitRequirement` and `updateRequirement` can remain as private helpers only if useful internally; public callers use `saveRequirement`.

`saveRequirement` has three paths:

1. `empty + ui_affected=true`
   - Validate non-empty `document_md`.
   - Require non-empty `pages`.
   - Set every page to `design_status: "pending"`.
   - Set status to `submitted`.
   - Write requirement, document, translations, baseline, and rules.

2. `submitted/active + ui_affected=true`
   - Validate non-empty `document_md`.
   - Merge only changed pages from `pages`.
   - Preserve existing pages not present in the input.
   - Remove pages listed in `remove_page_ids`.
   - Map `new -> pending`, `patch -> expired`, `rebuild -> expired`.
   - Preserve existing `design_id` for patch/rebuild pages when available.
   - Replace `navigation` with the full navigation list from input.
   - Set status to `submitted` if any page is pending or expired; otherwise `active`.
   - Write requirement, document, translations, baseline, and rules.

3. `ui_affected=false`
   - Validate non-empty `document_md`.
   - Do not modify pages, baseline pages, navigation, or translations.
   - Update document and rules only.
   - Set status to `active` if there are no pages or all pages are done; otherwise keep current status.

`assertPages` becomes conditional: pages are required only when `ui_affected=true`.

`getLatestRequirement(product_id)` returns the most recently updated non-archived requirement. If none exist, it throws `REQUIREMENT_NOT_FOUND`.

## Baseline Updates

Baseline updates no longer filter out `design_status: "expired"` pages. Patch and rebuild pages must update the functional baseline even though their designs require regeneration.

`mapNavigationToBaseline` also uses all requirement pages rather than filtering expired pages, so patch/rebuild navigation changes are mapped correctly.

`remove_page_ids` is the only deletion signal. Missing pages in a `save_requirement` input are treated as unchanged, not deleted.

## Rules

`packages/core` stores product-level BDD rules in:

```text
data/{product_id}/baseline/rules.yaml
```

Each persisted rule contains:

- `id`.
- optional `page_id`.
- `given`.
- `when`.
- `then`.
- `source_requirement`.

`rules[]` in `saveRequirement` means the complete rule set contributed by the current requirement, not a delta. On save:

- Remove all existing rules with `source_requirement === current requirement_id`.
- Rewrite incoming rule ids as `${requirement_id}-${ai_generated_id}`.
- Insert all incoming rules with `source_requirement` injected by the backend.
- Delete any `replaces_rule_id` targets.
- Delete any ids in `remove_rule_ids`.
- Delete page-associated rules for confirmed `remove_page_ids` when applicable.

AI conflict detection happens in the Agent command before `save_requirement`. The backend stores the confirmed result.

## Copy And Translations

Requirement pages and baseline pages store default-language copy:

```ts
interface CopyItem {
  context: string;
  text: string;
}
```

Translations are stored per requirement:

```text
data/{product_id}/{requirement_id}/copy-translations.yaml
```

The file contains page translation entries for non-default languages only. When source copy changes:

- New copy entries may receive new translations from Agent output.
- Deleted copy entries may remain in the translation file to avoid accidental data loss.
- Changed source text marks existing translation entries as `outdated: true` unless fresh translations are supplied.

`CopyService` owns reading, saving, merging, and updating translation files. `RequirementService` calls it inside the save transaction for UI-affecting paths.

## Transaction Model

Requirement saving uses rollback snapshots over the files touched by the selected path.

UI-affecting paths snapshot and write:

- `requirement.yaml`.
- `document.md`.
- `copy-translations.yaml`.
- `baseline.yaml`.
- `rules.yaml`.

Logic-only paths snapshot and write:

- `requirement.yaml`.
- `document.md`.
- `rules.yaml`.

Test hooks should cover failures after baseline, translations, document, and rules writes so rollback behavior is deterministic.

## Design Save Behavior

`DesignService` changes:

- `generate` remains for pages without existing designs.
- `refine` accepts normal done pages and also expired pages with existing designs when `change_type === "patch"`.
- `update` accepts normal done pages and also expired pages with existing designs when `change_type === "rebuild"`.
- `stageExistingDesign` accepts both `done` and `expired` pages when ownership and existing design metadata are valid.
- `getDesignMetadata(designId)` becomes public and returns absolute `pen_path`, `preview_path`, `version`, `created_at`, and `updated_at`.

Patch and rebuild both preserve the same `design_id` and increment design version. Their difference is Agent prompt behavior and `save_designs` mode, not the persistence history model.

`get_baseline_image` and the matching HTTP helper allow returning an expired page's existing preview so patch/rebuild flows can locate the old design.

## MCP Contract

Remove:

- `submit_requirement`.
- `update_requirement`.

Add:

- `save_requirement`.
- `get_product_rules`.
- `get_page_copy`.
- `update_page_copy`.

Change:

- `init_product_config` and `update_product_config` require `languages` and `default_language`.
- `get_product` returns the language fields.
- `get_requirement` returns a developer-facing aggregate: requirement fields, `document_md`, navigation, pages, structured copy, `copy_translations`, and design metadata when present.
- `help` returns the tool list plus a usage guide for AI developer workflows.
- `generate_page_design`, `generate_components`, and session selection check language completeness in addition to existing config gates.

The MCP layer aggregates `get_requirement` data to avoid a dependency cycle between `RequirementService`, `CopyService`, and `DesignService`.

## HTTP API

Add:

- `POST /api/products/:id/config`.
- `POST /api/products/:id/requirements/:reqId/save`.

Change:

- `POST /api/products/:id/requirements` only creates an empty requirement from `{ title }`.

## Agent Commands

The installable command set becomes:

- `fm-list-product`.
- `fm-status`.
- `fm-requirement`.
- `fm-design`.
- `fm-refine-design`.
- `fm-refine-components`.
- `fm-change-style`.
- `fm-rollback-design`.

Remove:

- `fm-upload-requirement`.
- `fm-update-requirement`.

`fm-requirement` flow:

1. Read current session.
2. Find the most recently updated non-archived requirement; block if none exist.
3. Read current requirement document, product baseline, product language config, and product rules.
4. Validate the user input as a product requirement. The check is permissive and only rejects unrelated content.
5. Use AI to merge, format, extract changed pages, infer `change_type`, produce structured copy, produce translations, produce rules, and produce explicit removal lists.
6. Use a second AI pass for conflict detection against baseline rules excluding the current requirement's old rules.
7. Show page changes and generated rules to the user for confirmation.
8. Apply confirmed overrides/removals and call `save_requirement`.

If `languages.length * page_count > 10`, the Agent splits AI output into one call for document/pages/navigation/rules/removals and a second call for translations.

`fm-list-product` handles missing `languages` during config completion and confirms the derived default language. Component generation prompts include the default UI language.

`fm-status` displays product language configuration for Claude, Codex, and Gemini templates.

`fm-design` uses page `change_type`:

- `new`: generate full design, save mode `generate`.
- `patch`: include old annotations and change summary, save mode `refine`.
- `rebuild`: regenerate full design while preserving design history, save mode `update`.

All Claude, Codex, and Gemini templates are updated together.

## Web Admin

Add lightweight i18n:

- `packages/web/src/i18n.ts`.
- `packages/web/src/LocaleContext.tsx`.

Supported locales are `en` and `zh`. The initial locale uses `localStorage.forma_locale` when present, otherwise `navigator.language`. Locale switching persists immediately and triggers React re-render through context.

`Layout` adds a compact `EN | 中` selector in the header.

Translate static UI strings through `t(key)`. Do not translate product names, product descriptions, style names, error codes, API paths, or technical identifiers.

`ProductNew`:

- Loads styles from `GET /api/styles`.
- Requires name, description, platform, style, languages, and default language.
- Derives default language as `en` when selected; otherwise the first selected language.
- Submits with `POST /api/products`, then `POST /api/products/:id/config`.
- If the config step fails after product creation, the product remains and can be completed from Product Detail.

`ProductDetail`:

- Shows platform, style, languages, default language, and config status.
- Offers a config completion form when config is incomplete.
- Creates requirements with title only.
- Displays empty requirements as pending content fill.

`BaselineView`:

- Renders structured copy instead of free-text copy.
- Shows a multilingual copy table when translation data is available.
- Highlights outdated translation entries.

## Risks And Controls

- AI-generated rules may be low quality. The Agent command must show generated rules to the user before saving.
- AI may misclassify `change_type`. The Agent command must show the change summary and let the user override it.
- Large AI outputs may exceed token limits. The Agent command uses the `languages.length * page_count > 10` split-output rule.
- Web translation coverage can be incomplete. Tests and grep review should cover static strings introduced in touched components.
- Schema-breaking changes can invalidate old local data. This is accepted by v0.3 because existing Forma home data is assumed non-production.
