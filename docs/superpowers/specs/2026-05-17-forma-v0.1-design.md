# Forma v0.1 Design

Date: 2026-05-17
Source: `design-version/DESIGN-v1.md`
Status: Approved design for implementation planning

## Goal

Implement Forma v0.1 as a full product-design asset management tool that turns requirement documents into Pencil-backed design artifacts, stores requirements/designs/baselines in `~/.forma`, exposes the data through MCP tools and a Web admin UI, and installs Agent commands for Claude, Codex, and Gemini.

## Scope

v0.1 includes:

- A pnpm workspace with separate packages for core logic, MCP server, Web API, Web UI, Agent templates, and CLI packaging.
- Real Pencil CLI integration for component generation, page design generation, export, validation, and rollback.
- `~/.forma` as the only runtime data location.
- Complete MCP tools listed in `DESIGN-v1.md`, including code-level hard gates.
- A Fastify-backed Web admin with product, requirement, baseline, style library, design preview, design history, diff, export, and annotation views.
- Complete LeaferJS-based annotation canvas for design inspection.
- Built-in 50+ `awesome-design-md` styles copied into `~/.forma/styles` on first use or install.
- Three-platform Agent command installation for Claude, Codex, and Gemini.

v0.1 does not include:

- Multi-user permissions.
- Project management tool integrations.
- Automatic online style sync.
- Advanced visual navigation graph rendering.
- Code generation from designs.
- A custom vector design engine.

## Architecture

Use `pnpm workspace` packages:

| Package | Responsibility |
| --- | --- |
| `packages/core` | Pure business logic: data paths, YAML persistence, IDs, status machines, baseline management, style indexing, Pencil CLI adapter, locks, `.pen` validation, annotation extraction, diff, install manifests. |
| `packages/mcp` | MCP stdio server using `McpServer`, `StdioServerTransport`, and `registerTool`; validates tool inputs with `zod` and delegates all logic to `core`. |
| `packages/server` | Fastify Web API and static Web asset serving; delegates all business rules and gates to `core`. |
| `packages/web` | React + Vite + Tailwind + LeaferJS admin UI; talks only to HTTP APIs. |
| `packages/agent` | Claude Markdown commands, Gemini TOML commands, Codex skill command templates, and shared Forma workflow guidance. |
| `packages/cli` | `forma` executable; routes `mcp`, `serve`, `install`, `uninstall`, `status`, and `version`, and packages web build output, styles, and Agent templates. |

The published user-facing CLI is still `forma`. The workspace split is an internal development and maintenance boundary.

## Data Model And Storage

All runtime data is under `~/.forma`. Tests may inject an alternate root.

```text
~/.forma/
├── config.yaml
├── session.yaml
├── pencil.lock
├── manifests/
├── skills/
├── commands/
├── data/
│   ├── products.yaml
│   └── P-xxxxxx/
│       ├── product.yaml
│       ├── baseline/baseline.yaml
│       └── R-xxxxxxxx/
│           ├── requirement.yaml
│           ├── document.md
│           └── D-xxxxxxxx/
│               ├── design.yaml
│               ├── design.pen
│               ├── design.v1.pen
│               └── preview@2x.png
├── library/
│   └── P-xxxxxx.lib.pen
└── styles/
```

IDs follow the source design:

- Product: `P-{6 hex chars}`
- Requirement: `R-{8 hex chars}`
- Design: `D-{8 hex chars}`

Requirement state machine:

```text
empty -> submitted -> active -> archived
```

Page design states:

```text
pending | done | expired
```

Baseline stores functional shape only: pages, features, interactions, navigation, copy, fields, and source requirements. Design/refine/change-style operations do not update baseline.

## Core Services

`packages/core` exposes focused services:

- `ProductService`: create/read products, complete platform/style config, verify config completeness.
- `RequirementService`: create empty requirements, submit requirement documents, update documents, mark affected pages, archive requirements.
- `BaselineService`: update functional baseline on requirement submit/update and remove baseline pages with no remaining source requirements.
- `DesignService`: save design outputs, manage versions, rollback, read previews, annotations, exports, and diffs.
- `StyleService`: install built-in styles, list styles, read `DESIGN.md`, validate extracted variables.
- `SessionService`: store current product only; latest requirement is calculated dynamically.
- `PencilService`: call real `pencil`, acquire locks, write temporary artifacts, validate outputs, export PNGs.
- `InstallService`: install/uninstall Agent command files and MCP configs with manifest-based cleanup.

YAML writes use temp-file-plus-rename. Requirement updates replace old data only after new data has been fully written. Pencil output is generated in `/tmp/forma-{uuid}` and moved into place after validation.

## Hard Gates

All hard gates are implemented in TypeScript, not in prompts. Agent command files may describe the workflow, but they cannot be the enforcement layer.

Required gates include:

- Product must exist.
- Product session requires `platform`, `style`, and `components_initialized`.
- Requirement submit requires latest/target requirement state `empty`, non-empty document, and non-empty pages.
- Requirement update requires state `submitted` or `active`, non-empty document, and atomic replacement.
- Design save requires all `page_id` values to belong to the requirement.
- Refine requires target page status `done`.
- Rollback requires design exists, page belongs to current/latest requirement, version is greater than 1, and previous `.pen` file exists.
- Pencil commands require CLI installed and authenticated.
- `.pen` files must parse as JSON, have non-empty children, and contain no recursive `"..."` truncation markers.
- Web API must enforce backend gates even when UI buttons are disabled.

Errors use stable error codes from `DESIGN-v1.md`, including `PRODUCT_CONFIG_INCOMPLETE`, `REQUIREMENT_STATUS_INVALID`, `DOCUMENT_EMPTY`, `PAGES_EMPTY`, `PAGE_NOT_OWNED`, `PEN_FILE_INVALID`, `PAGE_NOT_DONE`, `NODE_NOT_FOUND`, `VERSION_TOO_LOW`, `HISTORY_FILE_MISSING`, `PRODUCT_NOT_FOUND`, `REQUIREMENT_NOT_FOUND`, `DESIGN_NOT_FOUND`, `STYLE_NOT_FOUND`, `PENCIL_CLI_NOT_FOUND`, and `PENCIL_NOT_AUTHENTICATED`.

## Pencil Integration

Pencil integration is real in v0.1. Product code does not expose a user-facing mock mode.

Before any Pencil operation:

1. Run `pencil version`.
2. Run `pencil status`.
3. Acquire `~/.forma/pencil.lock`.
4. Generate into `/tmp/forma-{uuid}`.
5. Validate `.pen` and PNG outputs.
6. Move validated files into `~/.forma`.
7. Release lock and clean temp files.

The lock stores pid, acquired timestamp, operation, and product ID. Lock acquisition checks whether the pid is alive. Dead locks are reclaimed. Live locks older than 5 minutes are force-reclaimed.

Pencil generation modes:

- New page: `pencil --out ... --workspace ... --prompt ...`
- Expired incremental page: `pencil --in old.pen --out ... --workspace ... --prompt ...`
- Refine: `pencil --in old.pen --out ... --workspace ... --prompt ...`
- Components: `pencil --out components.lib.pen --prompt ...`
- Export: `pencil --in design.pen --export preview@2x.png --export-scale 2`

Prompt construction remains Agent-side for semantic work: conflict detection, page split, style variable extraction, and full-vs-incremental decision. MCP/core validates the submitted result and performs deterministic persistence.

## Agent Installation

`forma install --platform claude,codex,gemini` installs all three supported platforms in v0.1. This supersedes the `DESIGN-v1.md` MVP note that listed Claude-only support.

Install behavior:

- Generate 9 `fm-*` command files per selected platform.
- Install shared Forma workflow guidance under `~/.forma/skills/forma/SKILL.md`.
- Inject MCP server config for `forma mcp` where the platform supports MCP config.
- Back up any pre-existing target file before replacement.
- Write `~/.forma/manifests/{platform}.manifest`.

Uninstall behavior:

- Read manifest.
- Remove files that Forma installed.
- Restore backups when available.
- Remove Forma-owned MCP config entries without deleting unrelated user config.

The 9 commands are:

- `fm-list-product`
- `fm-status`
- `fm-upload-requirement`
- `fm-update-requirement`
- `fm-design`
- `fm-refine-design`
- `fm-refine-components`
- `fm-change-style`
- `fm-rollback-design`

## Web Admin UI

`packages/web` uses:

- React
- Vite
- Tailwind CSS v4 through `@tailwindcss/vite`
- `@import "tailwindcss"` in the main CSS file
- LeaferJS for annotation canvas

Pages:

- `/products`: product cards, config status, latest requirement status, create entry.
- `/products/new`: product name, description, optional platform, optional style.
- `/products/:productId`: requirement list, baseline summary, new requirement, archive controls.
- `/products/:productId/baseline`: functional page list and navigation list.
- `/products/:productId/requirements/:reqId`: requirement document, pages, designs, design history.
- `/products/:productId/requirements/:reqId/designs/:designId`: annotation canvas and property panel.
- `/styles`: style grid, search, and category filters.
- `/styles/:name`: style preview, `DESIGN.md`, and variables.

UI style is an operational workbench: restrained, dense enough for repeated use, with side navigation, top context, tables/lists, split panels, and direct error states. Agent-only actions are not added to Web UI.

## Annotation Canvas

The design annotation page has three layers:

1. 2x PNG base image.
2. Transparent node hit-test layer from parsed `.pen` nodes.
3. Dynamic annotation layer for hover boxes, selected-node outlines, dimensions, and spacing lines.

Interactions:

- Zoom and pan through LeaferJS viewport behavior.
- Hover a node to show border and size.
- Click a node to show full properties in the side panel.
- Copy node ID/path for `fm-refine-design`.
- Select two nodes to show spacing.
- Export selected node asset through the API.

The browser never reads `.pen` directly. It receives annotation JSON from the API.

## Styles

v0.1 includes the full `awesome-design-md` style set as package resources. First run or install copies styles into `~/.forma/styles`.

Each style has:

- `DESIGN.md`
- `preview@2x.png`
- entry in `styles.yaml`

The preview template is fixed and included as `_preview-template.pen`. Automatic online sync is deferred. Missing extracted style variables are filled by explicit defaults and validated by `core`.

Required default variables:

- `primary`
- `background`
- `text-primary`
- `font-heading`
- `font-body`
- `border-radius`
- `spacing-unit`

## CLI

`forma` commands:

- `forma mcp`
- `forma serve`
- `forma serve start`
- `forma serve stop`
- `forma install --platform claude,codex,gemini`
- `forma uninstall --platform claude,codex,gemini`

- `pnpm install` succeeds.

## Open Risks

- Real Pencil generation quality and latency are external to Forma. Forma mitigates this with validation, history, rollback, and refine workflows.
- Full 50+ style bundling increases package size. This is accepted to satisfy offline style availability.
- Three-platform command installation has platform-specific config edge cases. Manifest-based install/uninstall and tests reduce the risk.
- Annotation extraction depends on `.pen` structure stability. The parser should be isolated behind `DesignService`/annotation modules.
- The multi-package workspace raises build-order and packaging complexity. `packages/cli` must own final packaging checks.
