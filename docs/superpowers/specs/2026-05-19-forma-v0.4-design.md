# Forma v0.4 Design

Date: 2026-05-19
Source: `design-version/DESIGN-v4.md`
Status: Approved design for implementation planning

## Goal

Implement Forma v0.4 as one complete release batch. The release fixes MCP installation blockers, simplifies Agent product selection, moves component initialization to the design workflow, improves Web style visualization and admin polish, and adds safe product deletion with serialized product mutations and recoverable deletion transactions.

The implementation follows the full scope in `design-version/DESIGN-v4.md`; no feature area from that source design is intentionally deferred.

## Scope

v0.4 includes:

- Correct Claude MCP config writing under `mcpServers.forma`, including cleanup of historical top-level `forma` entries.
- Dynamic MCP command resolution during CLI install so local development works when `forma` is not on `PATH`.
- Simpler Agent product selection across Claude, Codex, and Gemini templates.
- Product session selection that requires product platform, style, languages, and default language, but not component initialization.
- `fm-design` component initialization fallback when components are missing during design generation.
- Web style visualization through richer style cards, DESIGN.md frontmatter parsing, a reusable `StylePreviewPanel`, and a platform-aware style picker in product creation.
- Web admin visual polish, loading skeletons, improved card hierarchy, navigation affordances, and empty states.
- Product deletion from Core, Server, MCP, Agent guidance, and Web, with explicit product ID confirmation.
- Product mutation locking for all product-scoped writes that can race with deletion or recovery.
- Recoverable deletion transactions using staging directories, `state.json`, backups, rollback, cleanup, and startup recovery.
- Documentation updates for MCP, Agent workflows, Web behavior, and deletion constraints.

v0.4 does not include:

- Requirement deletion as an API, MCP tool, or Web action.
- Soft delete, trash, batch delete, undo, or restore UI.
- New UI component libraries or frontend dependencies.
- New backend APIs beyond `DELETE /api/products/:id`.
- Changes to existing MCP tool API shapes except adding `delete_product` and changing `generate_components` to route through store-level orchestration.
- Dynamic Google Font loading in Web previews.
- YAML dependency changes for frontend DESIGN.md parsing.

## Architecture

v0.4 remains contract-first and keeps the existing monorepo package boundaries.

| Package | v0.4 Responsibility |
| --- | --- |
| `packages/core` | Own MCP install config writing, product config gates, product mutation locking, deletion transactions, recovery, store-level component generation orchestration, and shared error codes. |
| `packages/cli` | Resolve the runtime MCP command before constructing `InstallService`. |
| `packages/mcp` | Expose `delete_product`, wait for deletion recovery at startup, route `generate_components` through the store, and preserve structured error reporting. |
| `packages/server` | Expose product deletion, map product mutation lock conflicts to HTTP 409, and await deletion recovery in `onReady`. |
| `packages/web` | Render richer style previews, platform-aware style picking, visual loading states, and product deletion confirmation flows. |
| `packages/agent` | Update installed templates for product selection, design-time component initialization, and delete confirmation guidance. |
| `docs` / `README.md` | Document the changed MCP config, Agent workflow, product deletion behavior, and absence of requirement deletion. |

Core remains the source of truth for data consistency. Server and MCP are thin adapters. Web and Agent templates present the new behavior without making filesystem assumptions.

## Implementation Slices

The release is one batch, but the implementation plan should order work by risk:

1. Install and Agent workflow fixes.
2. Web style visualization and admin polish.
3. Product mutation lock and store-level orchestration.
4. Product deletion transaction and recovery.
5. Server, MCP, Web, Agent, and docs deletion integration.

This order gives useful focused test checkpoints while preserving one release scope.

## Install And MCP Command Resolution

`packages/core/src/install.ts` gains an `mcpCommand` option:

```ts
export interface InstallServiceOptions {
  formaHome?: string;
  userHome?: string;
  templatesDir?: string;
  mcpCommand?: { command: string; args: string[] };
}
```

The default remains `{ command: "forma", args: ["mcp"] }` so direct core callers and existing unit tests preserve behavior unless they opt in. The CLI passes the resolved command.

`packages/cli/src/index.ts` resolves MCP command synchronously:

- If `forma` is on `PATH`, use `{ command: "forma", args: ["mcp"] }`.
- Otherwise use `{ command: process.execPath, args: [packageCliEntrypoint(), "mcp"] }`.

All install targets use `this.mcpCommand`:

- Claude writes `mcpServers.forma`, not top-level `forma`.
- Gemini writes `mcpServers.forma`.
- Codex writes the managed TOML section with dynamic `command` and `args`.

Claude install, uninstall, and backup merge always remove the historical top-level `forma` entry. Legal user-owned `mcpServers.forma` entries from backups are preserved when backup recovery rules say they should be restored. If the current Claude config is missing and only a backup exists, recovery reads and sanitizes the backup instead of blindly copying it.

Codex TOML generation uses `JSON.stringify` for command and args so absolute paths with spaces, backslashes, or quotes remain valid TOML.

## Agent Product Selection And Design Initialization

`set_current_session` no longer requires `components_initialized`. Core session selection checks only:

- `platform`.
- `style`.
- `languages`.
- `default_language` is present and included in `languages`.

Agent templates for `fm-list-product` across Claude, Codex, and Gemini are simplified:

1. Call `list_products`.
2. Display a numbered list using only product name and product ID.
3. Ask the user to choose by number.
4. Call `set_current_session`.
5. If basic product config is incomplete, follow shared config completion guidance.
6. On success, fetch and display the latest requirement summary.

`fm-list-product` no longer triggers component generation or default-language confirmation for component generation.

Agent templates for `fm-design` become responsible for component initialization fallback. If design generation returns `PRODUCT_CONFIG_INCOMPLETE` with missing `components_initialized`, the Agent:

1. Confirms default language before continuing.
2. Calls `generate_components` with product config and default language in the prompt.
3. Calls `complete_product_init`.
4. Retries the design generation.

The template continues to inject exact structured page copy into design prompts and maps page `change_type` as `new -> generate`, `patch -> refine`, and `rebuild -> update`.

## Web Style Visualization

Web style visualization is implemented without new frontend dependencies.

`StyleCard` changes from a single color square to a compact preview strip using the existing seven metadata variables:

- `background`.
- `primary`.
- `text-primary`.
- `font-heading`.
- `font-body`.
- `border-radius`.
- `spacing-unit`.

Missing variables use stable fallbacks so style cards never become error states.

`packages/web/src/utils/parseDesignMd.ts` parses only DESIGN.md frontmatter. It extracts:

- `colors`.
- `typography`.
- `rounded`.
- `spacing`.
- `components`.

The parser is intentionally conservative:

- It supports 2-space indentation, object nesting, and scalar values.
- It supports keys containing letters, numbers, hyphens, underscores, and dots.
- It accepts quoted strings, unquoted strings, numbers, `transparent`, `0`, and token-like values such as `9999px`.
- It skips block scalars, arrays, complex structures, multiline strings, and unknown sections.
- It returns `warnings: string[]` instead of throwing for partial parse issues.
- It returns empty structures for styles with no frontmatter.

Token references such as `{colors.primary}` are resolved by the rendering layer, not by the parser.

`StylePreviewPanel` renders live style previews for `mobile`, `desktop`, `tablet`, and `web` using structured tokens first and metadata variables as fallback. It also renders color palette, typography scale, spacing, radius, and non-blocking parser warning copy when fallback behavior matters.

Template heuristics:

- Buttons prefer component names containing `button-primary`, then any component containing `button`, then fallback variables.
- Navigation prefers component names containing `nav`, then fallback variables.
- Cards use available rounded, spacing, and surface colors.
- Headings prefer `typography.display-lg` or `typography.title-md`.
- Body text prefers `typography.body-md`.

`StyleDetail` always shows `StylePreviewPanel`. Static server preview PNGs are additional content when available; they do not replace live variable previews.

`ProductNew` replaces the native style select with `StylePickerDialog`:

- The user must choose platform before opening the style picker.
- The closed state shows selected style summary or a disabled prompt.
- The dialog left pane provides search and candidate style cards.
- The dialog right pane shows `StylePreviewPanel` with `previewType` fixed to the selected platform.
- Selecting a style in the list changes only the candidate preview.
- The form value changes only when the user clicks the primary "use this style" action.
- `getStyle(name)` is loaded on demand and cached by style name.
- Load failure keeps metadata fallback preview visible and displays a non-blocking warning.

## Product Mutation Lock

`packages/core/src/product-mutation-lock.ts` implements one lock per Forma home:

```text
<formaHome>/tmp/locks/product-mutations.lock
```

Cross-process acquisition uses atomic `mkdir`. The lock directory contains `owner.json` with:

- `pid`.
- `operation`.
- `product_id`.
- `created_at`.
- `updated_at`.

Same-process callers use the same queue so two async mutations cannot race inside one process.

Fixed parameters:

- Stale threshold: `120000ms`.
- Wait timeout: `30000ms`.
- Retry interval: `100ms`.
- Jitter: up to `25ms`.
- Heartbeat: every `5000ms`.

If the owner process is gone or `updated_at` is stale, the lock may be removed and the event is recorded as a warning. If acquisition times out, Core throws `FormaError("PRODUCT_MUTATION_LOCKED", ...)` with details including operation, product ID, lock path, and waited milliseconds.

Stale-lock and release warnings are not allowed to disappear. Deletion and recovery paths merge product mutation warnings into `DeleteProductResult.recovery_warnings` or `ProductDeletionRecoveryResult.warnings`. Other public mutation paths call `onProductMutationWarning` when provided; when no sink is provided, they must use the default sink `process.emitWarning(warning, { type: "FormaProductMutationWarning" })`. Focused tests must cover at least one stale-lock warning path without a custom sink and assert that it is observable through Node's warning mechanism.

All product-scoped writes that can race with deletion or recovery are serialized through this lock. The required public mutation entry points are:

- `ProductService.createProduct`.
- `ProductService.initProductConfig`.
- `ProductService.markComponentsInitialized`.
- `SessionService.setCurrentProduct`.
- `RequirementService.createEmptyRequirement`.
- `RequirementService.submitRequirement`.
- `RequirementService.updateRequirement`.
- `RequirementService.saveRequirement`.
- `RequirementService.archiveRequirement`.
- `BaselineService.updateFromRequirement`.
- `CopyService.saveTranslations`.
- `CopyService.updatePageTranslations`.
- `CopyService.mergeTranslations`.
- `DesignService.saveDesigns`.
- `DesignService.rollbackDesign`.
- `DesignService.exportDesignAsset`.
- `store.generateComponents`.
- `store.deleteProduct`.
- `store.recoverPendingProductDeletes`.

Public mutation methods acquire the lock and then call internal locked helpers. Internal service-to-service calls use locked helpers so they do not attempt nested acquisition. For example, `RequirementService.submitRequirement`, `RequirementService.updateRequirement`, and `RequirementService.saveRequirement` acquire the lock once and then call `BaselineService.updateFromRequirementLocked` and `CopyService.saveTranslationsLocked`. `CopyService.updatePageTranslations` holds the same lock for the whole read-merge-write sequence, not only for the final save. `DesignService.rollbackDesign` calls internal file helpers while holding the lock. `store.deleteProduct` calls only internal locked deletion and recovery helpers.

Every relevant service options type accepts `productMutationLock?: ProductMutationLock` and `onProductMutationWarning?: (warning: string) => void`. If `productMutationLock` is omitted, the constructor uses `getProductMutationLock(home)` to obtain the same in-process lock instance for the same resolved Forma home. If `onProductMutationWarning` is omitted, the constructor uses the default `process.emitWarning` sink described above. This keeps direct service construction in tests and external core callers serialized without requiring every caller to manually pass a lock. `createFormaStore()` still constructs or obtains one lock and explicitly passes the same instance to all relevant services.

`getProductMutationLock(home)` is process-local and keyed by resolved `formaHome`. It must not create a different lock per service instance for the same home.

`createFormaStore()` exposes:

- `runProductMutation(...)`.
- `generateComponents(...)`.
- `deleteProduct(...)`.
- `recoverPendingProductDeletes(...)`.

The store-level API includes:

```ts
export interface ComponentGenerator {
  generateComponents(input: GenerateComponentsInput): Promise<GeneratedComponents>;
}

export interface ProductMutationLock {
  run<T>(
    input: { operation: string; product_id?: string },
    fn: (context: ProductMutationContext) => Promise<T>
  ): Promise<T>;
}

export function getProductMutationLock(home: string): ProductMutationLock;

export interface FormaStoreOptions {
  home: string;
  bundledStylesDir?: string;
  syncStyleLimit?: number;
  pencilService?: ComponentGenerator;
  productMutationLock?: ProductMutationLock;
  onProductMutationWarning?: (warning: string) => void;
}

type ProductDeletionRecoveryResult = {
  recovered: number;
  cleaned: number;
  warnings: string[];
};

type ProductMutationContext = {
  operation: string;
  product_id?: string;
  warnings: string[];
};

type GenerateComponentsInput = {
  product_id: string;
  prompt: string;
  workspace: string;
};

recoverPendingProductDeletes(): Promise<ProductDeletionRecoveryResult>;
generateComponents(input: GenerateComponentsInput): Promise<GeneratedComponents>;
runProductMutation<T>(
  input: { operation: string; product_id?: string },
  fn: (context: ProductMutationContext) => Promise<T>
): Promise<T>;
```

`FormaStoreOptions.pencilService` is a required adapter seam for tests and alternate low-level component-generation implementations. Its type is structural and contains only `generateComponents`; it must not be typed as `PencilService`, because `PencilService` has private fields and would block simple test mocks. The injected component generator is private to the store and may be used only by `store.generateComponents()`. It must not affect `SyncService`, page design generation, style sync, or any other low-level Pencil operation.

Public mutation methods must not be called from inside a `runProductMutation()` callback that already holds the product mutation lock. Internal code inside that callback must use locked helpers, or the lock implementation must explicitly support same-context reentry. The implementation plan should prefer locked helpers so same-process queue deadlocks cannot occur by accident.

## Component Generation Store Boundary

`store.generateComponents(input)` is the only business entry point allowed to write `library/<productId>.lib.pen`.

Its internal flow is fixed:

1. `runProductMutation({ operation: "generate_components", product_id })`.
2. `products.getProduct(product_id)`.
3. `assertProductConfig(product, product_id, ["platform", "style", "languages"])`.
4. Call the low-level Pencil adapter to generate the component library.

`PencilService` stays a low-level Pencil adapter. If `PencilService.generateComponents` remains public on the class, only store-owned code may call it for business component generation. MCP, Server, Agent tools, and Web-facing adapters must not call it directly.

`createFormaStore()` must accept `pencilService` through `FormaStoreOptions` as a `ComponentGenerator`. That injected generator is consumed only by `store.generateComponents()`, and component library writes still go through the store method. `CreateFormaToolsOptions.pencil` must not affect `generate_components`. Tests that need to replace component generation should either inject `pencilService` into `createFormaStore({ pencilService })` or use a mock store with a mocked `generateComponents`.

No component library write path may bypass `store.generateComponents()`.

## Product Deletion Transaction

Product deletion is exposed only through `store.deleteProduct(input)`.

Input:

```ts
type DeleteProductInput = {
  product_id: string;
  confirm_product_id: string;
};
```

Result:

```ts
type DeleteProductResult = {
  product_id: string;
  deleted: true;
  session_cleared: boolean;
  cleanup_pending: boolean;
  recovery_warnings: string[];
};
```

Deletion state:

```ts
type ProductDeletionState = {
  schema_version: 1;
  operation_id: string;
  product_id: string;
  created_at: string;
  updated_at: string;
  committed: boolean;
  phase: "created" | "backed_up" | "moved" | "index_written" | "session_written" | "committed";
  backups: {
    products_yaml: "backups/products.yaml";
    session_yaml?: "backups/session.yaml";
  };
  moved_paths: Array<{
    kind: "product_data" | "component_library";
    original_path: string;
    staged_path: string;
    required: boolean;
  }>;
  missing_paths: string[];
  session_was_current: boolean;
  warnings: string[];
};
```

`original_path` is relative to `formaHome`. `staged_path` is relative to the deletion operation directory. `state.json` is written atomically after every phase change, and recovery decisions must be based on this schema, not on guessed directory layout.

Pure validation happens before lock acquisition. Missing or mismatched `confirm_product_id` throws `INVALID_INPUT` and must not read, recover, lock, or write.

After validation, deletion runs inside the product mutation lock:

1. Recover pending deletions.
2. Verify the product exists.
3. Create `tmp/deletions/<operationId>/`.
4. Write initial `state.json` with `phase = "created"`.
5. Back up `data/products.yaml` and optional `session.yaml`.
6. Update state to `phase = "backed_up"`.
7. Move `data/<productId>/` and optional `library/<productId>.lib.pen` into staging.
8. Update state to `phase = "moved"`.
9. Remove the product from `data/products.yaml`.
10. Update state to `phase = "index_written"`.
11. If the session pointed at the product, write `{ current_product: null }`.
12. Update state to `phase = "session_written"` when applicable.
13. Mark `committed = true` and `phase = "committed"`.
14. Remove the staging operation directory.

Deletion moves the whole `data/<productId>/` directory so current and future product-scoped data are covered. It also stages the component library file when present.

Rollback uses only `state.json`. It restores backed-up index and session files and moves staged paths back to their original relative paths. Cleanup after a committed transaction removes staging. Cleanup failure returns `cleanup_pending: true`; it does not roll back the committed active state.

Phase failure handling:

- Pure input validation failure: do not acquire the lock, recover pending deletions, read product files, or write anything; return `INVALID_INPUT`.
- Lock acquisition failure: do not recover or write anything; return `PRODUCT_MUTATION_LOCKED`.
- Pending recovery failure before the new delete: record warnings and continue only if no unrecovered uncommitted staging for the same product would be overwritten.
- Product-not-found after recovery: do not create staging or modify active files.
- Operation directory creation failure: remove the incomplete operation directory if present and do not modify active files.
- Backup, move, index write, or session write failure: use `state.json.phase` to restore `products.yaml`, restore `session.yaml` when backed up, move staged product data and component library paths back, then rethrow.
- Staging cleanup failure after commit: do not roll back; return `cleanup_pending: true` and leave the operation directory for later cleanup.

`recoverPendingProductDeletes()` scans `tmp/deletions`:

- `committed !== true`: attempt recovery.
- `committed === true`: attempt cleanup.

Warnings are returned and never silently swallowed.

## Server Contract

Server adds:

```http
DELETE /api/products/:id
```

Body:

```json
{ "confirm_product_id": "P-123abc" }
```

Success:

```json
{
  "product_id": "P-123abc",
  "deleted": true,
  "session_cleared": true,
  "cleanup_pending": false,
  "recovery_warnings": []
}
```

Errors:

- Missing or mismatched confirmation returns HTTP 400 with `error_code = "INVALID_INPUT"`.
- Missing product returns the existing not-found shape with `PRODUCT_NOT_FOUND`.
- Product mutation lock timeout returns HTTP 409 with `PRODUCT_MUTATION_LOCKED`.

`buildServer()` registers an `onReady` hook that awaits `store.recoverPendingProductDeletes()`. Startup fails if recovery throws. Recovery warnings are logged.

## MCP Contract

MCP adds `delete_product` and does not add `delete_requirement`.

MCP server options include a minimal logger seam for recovery warnings:

```ts
interface FormaMcpLogger {
  warn(input: { warning: string }, message: string): void;
}

interface CreateFormaMcpServerOptions extends CreateFormaToolsOptions {
  home?: string;
  bundledStylesDir?: string;
  logger?: FormaMcpLogger;
}
```

If `logger` is omitted, startup recovery warnings are written to stderr.

Input:

```json
{ "product_id": "P-123abc", "confirm_product_id": "P-123abc" }
```

Output:

```json
{
  "product_id": "P-123abc",
  "deleted": true,
  "session_cleared": true,
  "cleanup_pending": false,
  "recovery_warnings": []
}
```

The `delete_product` input schema must validate both fields and refine that `confirm_product_id === product_id`. Missing or mismatched confirmation is an MCP tool-input failure and returns the existing wrapper error code `VALIDATION_ERROR`. Runtime Core failures preserve Core error codes.

`createFormaMcpServer()` becomes async and returns `Promise<McpServer>`. It creates the store, awaits deletion recovery, records any successful recovery warnings to stderr or an injected logger, registers tools, and returns the server. It does not connect the transport. `main()` awaits `createFormaMcpServer(options)`, then creates and connects the stdio transport. If recovery fails, `createFormaMcpServer()` rejects and `main()` never connects stdio.

`generate_components` calls `store.generateComponents(input)` and cannot bypass store locking through `new PencilService(...)`.

## Agent Delete Guidance

Shared Agent guidance requires explicit confirmation before `delete_product`:

- Repeat product name.
- Repeat product ID.
- Describe deletion scope.
- Ask for explicit confirmation.
- Pass the same product ID as both `product_id` and `confirm_product_id`.

If `session_cleared` is true, the Agent tells the user to run `fm-list-product` again. If `recovery_warnings` is non-empty, the Agent summarizes the warnings instead of reporting only success.

## Web Product Deletion

Web adds deletion entry points in:

- Product list cards.
- Product detail Danger zone.

Both use `ConfirmDeleteDialog`, not `window.confirm`.

The dialog:

- Shows product name, product ID, and deletion scope.
- Requires the user to type the exact product ID.
- Enables the final danger action only after exact match.
- Treats Esc and close as cancel only.

After success:

- Product list removes the product and refreshes summary state.
- Product detail navigates to `/products`.
- `session_cleared` shows a short notice that product selection was cleared.
- `cleanup_pending` shows a warning.
- `recovery_warnings` are summarized visibly.

## Errors

Core adds:

- `INVALID_INPUT` for pure core input validation failures such as mismatched delete confirmation.
- `PRODUCT_MUTATION_LOCKED` for lock acquisition timeout.

Server keeps `RouteInputError` for HTTP input validation and returns `error_code = "INVALID_INPUT"`. MCP keeps `VALIDATION_ERROR` for schema/refine failures and passes Core errors through for runtime failures.

## Documentation

Update:

- `docs/MCP.md` for `delete_product`, no `delete_requirement`, `generate_components` store orchestration, and product config completeness semantics.

Key acceptance criteria:

- Claude install writes `mcpServers.forma` and cleans top-level `forma`.
- MCP config command resolves to an executable command when local `forma` is not globally installed.
- Product selection succeeds without `components_initialized` when platform, style, languages, and default language are complete.
- `fm-design` owns missing component initialization recovery.
- Style previews render from structured DESIGN.md tokens when available and from metadata fallback otherwise.
- Product creation style picker only commits the selected style after explicit confirmation.
- Loading states use skeletons where page structure is known.
- Product deletion requires matching product ID confirmation.
- Pure delete input validation does not acquire locks, run recovery, or touch files.
- Product mutation lock serializes all product-scoped writes that can race with deletion or recovery.
- Deletion failure before commit restores index, product directory, component library, and session.
- Cleanup failure after commit returns `cleanup_pending: true`.
- Server and MCP startup await deletion recovery.
- MCP exposes `delete_product` and does not expose `delete_requirement`.
- Web, Server, MCP, and Agent surface recovery warnings.
