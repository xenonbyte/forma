# Forma route: fm-brand-assets

Use shared Forma guidance at ~/.forma/skills/forma/SKILL.md.

Generate a product's marketing brand assets — **store shots** (app-store / sharing screenshots), an optional **banner**, and **posters** (vertical / landscape / square layouts) — as HTML compositions that Forma core renders to PNG through the localize + sandbox render path. Store shots composite a device frame + the product's real page screenshot + selling copy on a brand-colored background; posters are full-bleed layouts with optional generated illustration material. You author the HTML and the prompts; Forma core renders and persists. These are product-level marketing assets.

A bare `fm-brand-assets` rerun (no description) = FULL regeneration of every planned brand asset. Before executing a bare rerun, double-confirm with the user that they want to regenerate the marketing assets.

Preconditions (all fail = STOP + guidance; never silently proceed):
- A product must exist and be selected. If absent, guide the user to run `fm-list-product` first (and to create/select a product if none exists).
- The product must be configured (`platform` + `brand_style`). If either is unset, STOP here (before calling any tool) and guide the user to run `init_product_config` (or the backstage config) before retrying. The plan (sizes / surfaces / counts) is derived from `platform`, so this is required. (This is a template-layer STOP; no tool error code is produced because no tool is called.)
- An image model must be configured (HARD). `generate_image` produces the background / illustration material. There is no tool that reports the media config, so the precondition surfaces from the generator: if `generate_image` returns `MEDIA_NOT_CONFIGURED`, STOP and tell the user to configure an image model in the web **Settings →「图片模型」(Image model)** section, then re-run. Do NOT retry blindly or fall back to a placeholder.
- An app icon must already exist (HARD). Call `list_brand_assets(product_id, kind="app-icon")`. If none exists, STOP and guide the user to run `fm-app-icon` first — store shots always show the icon. Do NOT proceed without it.
- STORE SHOTS additionally need at least one design artifact WITH a preview (HARD, store-shots only). Call `list_product_artifacts(product_id)` and confirm a design-page / component-library artifact with a preview exists. If none does, STOP for the store-shot path and guide the user to run `fm-design` first (store shots embed a real page screenshot). POSTERS do NOT need design previews.

Execution:
1. Require product_id from context or ask the user to run `fm-list-product` first.
2. Read the product config (platform + brand_style). If `brand_style` or `platform` is unset, STOP and guide to `init_product_config` (see preconditions). Confirm with the user which asset(s) to produce, and double-confirm on a bare rerun.
3. Run the preconditions: `list_brand_assets(product_id, kind="app-icon")` (app icon HARD); for store shots, `list_product_artifacts(product_id)` to confirm a design artifact WITH a preview exists (design-preview HARD). STOP + guide on any failure as above.
4. **Read the plan FIRST — do not hardcode sizes/counts/surfaces.** Call `get_brand_asset_plan(product_id)`. It returns `{ productId, platform, surfaces, entries:[{ kind, surface?, variant?, width, height, count, ... }] }`. Each entry is one thing to produce: store-shots carry `surface` (android/ios for mobile/tablet) and `count`; an optional `banner` entry appears only when enabled (per surface); poster entries carry `variant` (portrait/landscape/square) and NO surface. You MUST iterate the plan's `entries` and use each entry's exact `{width,height}` (and `surface`/`variant`) — it is FORBIDDEN to invent or hardcode dimensions/counts that did not come from the plan.
5. Read the materials: `get_style(brand_style)` for the brand tokens (resolved hex + roles), the app icon via the brand ref `forma-image://brand/app-icon`, and (store shots / banner) the design previews from `list_product_artifacts` / `get_product_artifact` for the real page screenshot. **Embed the page screenshot as a `data:` URI** (e.g. `data:image/png;base64,...`) directly inside the composed HTML — read the preview PNG bytes via `get_product_artifact` / `export_artifact` and base64-inline them. The render sandbox rejects remote URLs (http/https); there is no `forma-image://` namespace for a product page preview (only `forma-image://brand/app-icon` and generated `forma-image://<uuid>` staging refs are valid). A `data:` URI is the only sandbox-allowed way to embed the product's own page screenshot.
6. **For EACH plan entry, compose HTML and persist via the discriminated-union save.** Compose layout HTML for the entry (a device frame + the product's real page screenshot + short selling copy + a brand-colored background for store-shot/banner; a full-bleed branded composition for poster). Optionally generate background/illustration material: `generate_image(product_id, purpose="store-shot-bg" | "poster-bg" | "illustration", count=3)` built from the matching scaffold in `craft/image-prompts.md` (inject the brand palette), then **veto** — Read each candidate's `preview_path` and apply the veto checklist + the per-purpose veto; reference the chosen `forma-image://<uuid>` as the HTML background. The HTML references ONLY `forma-image://` (the brand icon and any generated background) and the product's own page preview (as a `data:` URI) — no remote refs (the render sandbox whitelist rejects them). Persist each entry with:
   `save_brand_asset(product_id, kind=<entry.kind>, name=<derived>, source={ html }, surface=<entry.surface? if present>, variant=<entry.variant? if present>, target={ width: entry.width, height: entry.height })`.
   The save is a discriminated union on `kind`; for these media kinds you pass `name` + `source.html` + `target` (+ optional `surface`/`variant`). `brand_style` comes from the product config — do NOT pass it. It returns `{ kind, asset:{ name, files:[{path,width,height}], surface?, variant?, generated_at, warnings } }`. Produce one save per plan entry × `count`: store-shots = one per surface (android + ios for mobile/tablet) × the entry's `count`, with a distinct `name` each (e.g. `android-1`, `ios-2`); banner = one per surface when enabled; posters = one per enabled style (portrait/landscape/square), no surface.
7. Report the saved files (`asset.files[]` paths + dimensions, grouped by kind then surface) and the brand-assets canvas URL `/products/<product_id>/brand-assets` where the store-shot / banner / poster groups render (the groups appear dynamically as their data lands). Note the export hint: the canvas's "export all" / the brand-assets export endpoint zips every brand-asset file for the product. Report stable error codes exactly as returned. The real codes reachable in this flow: `MEDIA_NOT_CONFIGURED` (image model not configured, from `generate_image`); `PRODUCT_NOT_FOUND` (from `get_brand_asset_plan` / other tools when the product id is unknown); `BRAND_ASSET_INVALID_INPUT` (from `save_brand_asset`, e.g. `details.reason: "product_not_configured"` when brand_style is missing).

The plan reader is `get_brand_asset_plan`; the brand-asset save tool is `save_brand_asset`; the image generator is `generate_image`. To remove a stale asset record + its files, `delete_brand_asset(product_id, kind, name)`.
