# Forma route: fm-brand-assets

Codex route: `$fm-brand-assets`.

Use shared Forma guidance at ~/.forma/skills/forma/SKILL.md.

Generate a product's marketing brand assets — **store shots** (app-store / sharing screenshots) and **posters** (vertical social / launch posters) — as HTML compositions that Forma core renders to PNG through the localize + sandbox render path. Store shots composite a device frame + the product's real page screenshot + selling copy on a brand-colored background; posters are a 1080×1920 vertical layout with optional generated illustration material. You author the HTML and the prompts; Forma core renders and persists. These are product-level marketing assets.

Preconditions (all fail = STOP + guidance; never silently proceed):
- A product must exist and be selected. If absent, guide the user to run `$fm-list-product` first (and to create/select a product if none exists).
- The product must be configured (`platform` + `brand_style`). If either is unset, STOP here (before calling any tool) and guide the user to run `init_product_config` (or the backstage config) before retrying. The store-shot preset set is derived from `platform`, so this is required. (This is a template-layer STOP; no tool error code is produced because no tool is called.)
- An image model must be configured (HARD). `generate_image` produces the background / illustration material. There is no tool that reports the media config, so the precondition surfaces from the generator: if `generate_image` returns `MEDIA_NOT_CONFIGURED`, STOP and tell the user to configure an image model in the web **Settings →「图片模型」(Image model)** section, then re-run. Do NOT retry blindly or fall back to a placeholder.
- An app icon must already exist (HARD). Call `list_brand_assets(product_id, kind="app-icon")`. If none exists, STOP and guide the user to run `$fm-app-icon` first — store shots always show the icon. Do NOT proceed without it.
- STORE SHOTS additionally need at least one design artifact WITH a preview (HARD, store-shots only). Call `list_product_artifacts(product_id)` and confirm a design-page / component-library artifact with a preview exists. If none does, STOP for the store-shot path and guide the user to run `$fm-design` first (store shots embed a real page screenshot). POSTERS do NOT need design previews.

Execution:
1. Require product_id from context or ask the user to run `$fm-list-product` first.
2. Read the product config (platform + brand_style). If `brand_style` or `platform` is unset, STOP and guide to `init_product_config` (see preconditions). Confirm with the user which asset(s) to produce: store shots, posters, or both.
3. Run the preconditions: `list_brand_assets(product_id, kind="app-icon")` (app icon HARD); for store shots, `list_product_artifacts(product_id)` to confirm a design artifact WITH a preview exists (design-preview HARD). STOP + guide on any failure as above.
4. **Query the presets — do not hardcode preset names; use the ids returned by `list_store_shot_presets`.** Call `list_store_shot_presets(product_id)` to get the available store-shot preset ids for THIS product's platform. It returns `{ presets: [{ id, width, height, source, verified_at }] }`. You MUST pass one of these RETURNED `id`s to `save_brand_asset(target={preset})`; it is FORBIDDEN to invent or hardcode a preset id/name that was not returned by this query.
5. Read the materials: `get_style(brand_style)` for the brand tokens (resolved hex + roles), the app icon via the brand ref `forma-image://brand/app-icon`, and (store shots) the design previews from `list_product_artifacts` / `get_product_artifact` for the real page screenshot. **Embed the page screenshot as a `data:` URI** (e.g. `data:image/png;base64,...`) directly inside the composed HTML — read the preview PNG bytes via `get_product_artifact` / `export_artifact` and base64-inline them. The render sandbox rejects remote URLs (http/https); there is no `forma-image://` namespace for a product page preview (only `forma-image://brand/app-icon` and generated `forma-image://<uuid>` staging refs are valid). A `data:` URI is the only sandbox-allowed way to embed the product's own page screenshot.
6. **STORE SHOTS — one per returned preset.** Compose layout HTML: a device frame + the product's real page screenshot + short selling copy + a brand-colored background. Optionally generate background material: `generate_image(product_id, purpose="store-shot-bg", count=3)` built from the **`store-shot-bg` scaffold in `craft/image-prompts.md`** (inject the brand palette), then **veto** — Read each candidate's `preview_path` and apply the veto checklist + the `store-shot-bg` per-purpose veto (reject if too busy to sit behind a device, if it already contains a phone/screenshot, or if it contains any text); reference the chosen `forma-image://<uuid>` as the HTML background. The HTML references ONLY `forma-image://` (the brand icon and any generated background) and the product's own page preview — no remote refs (the render sandbox whitelist rejects them). Persist each: `save_brand_asset(product_id, kind="store-shot", source={ html }, target={ preset: <a preset id from list_store_shot_presets> }, name)`.
7. **POSTERS — vertical 1080×1920.** Compose layout HTML for a 1080×1920 vertical poster. Optionally generate illustration material: `generate_image(product_id, purpose="poster-bg", count=3)` (or `purpose="illustration"`) built from the **`poster-bg` (or `illustration`) scaffold in `craft/image-prompts.md`**, then **veto** — Read each `preview_path` and apply the veto checklist + the `poster-bg` per-purpose veto (reject if the text-safe bands are unusable or it bakes in title text); reference the chosen `forma-image://<uuid>` in the HTML. Persist: `save_brand_asset(product_id, kind="poster", source={ html }, target={ width: 1080, height: 1920 }, name)`. The poster uses an explicit `{width,height}` target (1080×1920 is the poster spec), NOT a preset.
8. Report the saved files (`files[]` paths + dimensions) and the brand-assets canvas URL `#/products/<product_id>/brand-assets` where the store-shot and poster groups render (the groups appear dynamically as their data lands). Note the export hint: the canvas's "export all" / the brand-assets export endpoint zips every brand-asset file for the product. Report stable error codes exactly as returned. The real codes reachable in this flow: `MEDIA_NOT_CONFIGURED` (image model not configured, from `generate_image`); `PRODUCT_NOT_FOUND` (from `list_store_shot_presets` / other tools when the product id is unknown); `BRAND_ASSET_INVALID_INPUT` (from `save_brand_asset`, e.g. `details.reason: "product_not_configured"` when brand_style is missing, or an unknown preset id).

The preset query tool is `list_store_shot_presets`; the brand-asset save tool is `save_brand_asset`; the image generator is `generate_image`.
